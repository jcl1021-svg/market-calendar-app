# 美盘事件日历（Market Calendar）设计文档

日期：2026-07-02

## 一句话概述

一个托管在 GitHub Pages 上的深色月历网页，由 GitHub Actions 每天自动从**官方权威源**抓取数据、生成 `data.json`。用户打开网址即可查看非农、CPI、FOMC 和 CME（ES/NQ）假期，支持上/下月切换。

## 目标与约束

- **零维护**：配置一次后自动运行，用户不手动维护数据。
- **数据权威、真实**：只用官方公布的日程，不做规则推算。
- **视觉借鉴 TPT 后台日历**：深色主题，周末标红，工作日深绿，事件用橙色胶囊显示（**不要喇叭图标**），点某天弹窗看详情。
- **范围聚焦**：只显示四类事件——非农（NFP）、CPI、FOMC、CME 假期（全休 + 提前收盘）。

## 数据源（均为权威源）

实测发现 **BLS 与 CME 官网都封锁程序抓取（403 / Access Denied）**，云端 IP 更是必封，因此改走「权威且可自动获取」的路线。每个源在脚本中**互相独立**：某个源失败不影响其他源。

| 事件 | 采用的权威源 | 说明 |
|------|--------------|------|
| FOMC | 美联储官网 FOMC 日程页 (federalreserve.gov/monetarypolicy/fomccalendars.htm) | 解析各年会议日；声明发布日 = 会议第二天 14:00 ET；页面可抓 |
| 非农（NFP）、CPI | **FRED API**（圣路易斯联储，记录 BLS 官方发布日，含未来排期，返回 JSON） | 需免费 API key（存本地 `.env`，见下）；发布时间固定 8:30am ET |
| CME 假期（ES/NQ） | 脚本内**核准过的显式表** `CME_HOLIDAYS`（逐年逐条核对） | 全休 + 提前收盘 + 假期名；非推算，每年刷新时新增下一年并核对 |

### 已知风险（如实记录）

1. **BLS / CME 官网封锁抓取**：故不直接爬，改用 FRED（非农/CPI）与显式假期表（CME）。
2. **`exchange_calendars` 库不可靠**：实测漏掉 Juneteenth（6/19），故 CME 假期改用逐条核对的显式表。
3. **提前收盘精确时间**：约 13:00 ET；页面仅标"提前收盘"，不写精确时间（提前约两周才最终敲定）。

## 配色

最终采用 **TPT 原味**（用户测色提取的精确值）：工作日绿 `#1b332c`、周末/全休鲑 `#e0706c`、非本月暗红 `#764142`、今日绿框 `#4aa06a`。事件胶囊：经济=铁锈 `#a34f3a`、提前收盘=金 `#c8912f`、全休=深红 `#5e2b2b`。**全休日格子涂鲑色**（= 市场休市，TPT 本意）。铁锈/金 已过色盲验证（ΔE 31.8）。弹窗标题含具体节日名。切换配色只需改 `style.css` 的 `:root`。

## 架构

### 仓库结构

```
market-calendar/
├── index.html                    # 月历页面
├── style.css                     # 深色主题样式（借鉴 TPT）
├── app.js                        # 读取 data.json 渲染日历 + 弹窗 + 月份切换
├── data.json                     # 自动生成的数据（覆盖全年）
├── scripts/
│   └── fetch_data.py             # 抓官方源、生成 data.json
└── .github/
    └── workflows/
        └── update.yml            # 定时任务（每天）
```

### 数据管道（`fetch_data.py`）

1. 分别抓取 FOMC、BLS（非农/CPI）、CME 假期三个官方源。
2. 每个源独立 try/except，失败仅跳过该源并记录，不影响整体。
3. 统一整理为事件列表，写入 `data.json`。

### `data.json` 格式

```json
{
  "generated": "2026-07-02",
  "events": [
    {"date": "2026-07-02", "name": "Non-Farm Employment Change", "time": "8:30am ET", "type": "economic", "desc": "..."},
    {"date": "2026-07-03", "name": "Early Close (CME)",          "time": "~13:15 ET", "type": "early_close", "desc": "..."},
    {"date": "2026-12-25", "name": "Christmas — Market Closed",  "type": "holiday",    "desc": "..."}
  ]
}
```

- `type` 取值：`economic`（经济数据）、`holiday`（全休）、`early_close`（提前收盘）。
- 官方源给的是全年日程，因此 `data.json` 覆盖全年；页面默认显示当月，可切换任意月份。

### 部署与更新（方案 A：权威快照）

- 这些日期年初一次性公布、极少变，故**不用每日抓取**。GitHub Pages **纯静态托管**即可，不跑任何 Action。
- **更新方式**：约一年一次，本地跑 `python3 scripts/fetch_data.py` 重新生成 `data.json`，再 push。
- FRED key 存本地 `.env`（`.gitignore` 忽略），**只在本地生成时使用，绝不进公开仓库、不进网页**。

## 开发流程（重要）

**最终部署目标是 C（GitHub Pages）**，但为避免"每改一点就 push"的麻烦，开发期全程在本地预览：

1. 本地跑一次 `fetch_data.py` 生成 `data.json`；
2. 用极简静态服务器预览（`python3 -m http.server`，非会自动抓数据的本地服务），浏览器打开 `localhost` 反复调样式；
3. 全部满意后，**最后一步**才建 GitHub 仓库、开启 Pages、配置 Actions。

代码完全一致，本地验证与线上部署共用同一套文件，不浪费。

> 注：本地用 `file://` 直接打开 `index.html` 会因 CORS 无法 `fetch` 本地 `data.json`，因此预览必须经由静态服务器。

## 前端页面（借鉴 TPT 视觉）

- **布局**：深色底，Sun–Sat 月历网格。
- **月份切换条**：顶部 `<  July 2026  >`，月名居中、左右箭头，点箭头切上/下月（数据在本地，秒切）。
- **格子配色**：
  - 周末（Sat/Sun）：红/鲑色；
  - 工作日：深绿；
  - 今天：绿色边框高亮；
  - 全休假期格子：单独配色区分（与普通事件区分）。
- **事件胶囊**：橙色小胶囊，**只写事件名**（过长截断，不带喇叭图标）。
- **弹窗**：点某天弹出，显示日期、事件名、时间（ET）、说明——与 TPT 弹窗一致。

## 明确不做（YAGNI）

- 不做手动数据维护界面。
- 不做预测值/实际值/前值（官方日程不含这些，页面也不需要）。
- 不做除四类以外的事件（PPI、JOLTS、PCE、GDP 等）。
- 不做多市场/多时区切换（只 ES/NQ，时间统一 ET）。

## 测试要点

- `fetch_data.py`：对每个源用固定的样例 HTML/输入验证解析结果正确；验证某源失败时其余源仍产出数据。
- `data.json` 结构：字段完整、日期格式合法、type 取值合法。
- 前端：给定样例 `data.json`，验证当月渲染、周末/今天/假期配色、事件胶囊、弹窗内容、上/下月切换正确。
